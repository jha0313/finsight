Supabase MCP · github.com/supabase/mcp

get_advisors
DB가 스스로 떼주는 진단서

MCP는 Claude가 Supabase에 손을 뻗게 해주는 "콘센트"고, get_advisors는 그 콘센트로 누르는 "DB 건강검진 버튼"입니다. 누르면 DB가 알아서 보안·성능을 스캔하고 진단서를 돌려줍니다 — 고쳐주진 않고, 찾아줍니다.

01 / 큰 그림

Claude는 어떻게 내 DB까지 닿나

Claude가 DB 비밀번호를 직접 들고 있는 게 아니라, MCP 서버라는 중간 창구에 OAuth로 로그인하고 → 창구가 대신 DB를 만집니다.

🤖 Claude Code 클라이언트 / 에이전트
① get_advisors 호출 ④ 진단서(JSON) ◀
🔌 Supabase MCP mcp.supabase.com/mcp · OAuth 2.1
② project_ref 로 범위 ③ 진단 결과 ◀
🗄️ 내 Postgres finsight DB
🔒 창구는 제한할 수 있다. read_only=true 면 읽기만, project_ref 로 특정 프로젝트만, 기능 묶음(feature group)도 켠 것만 노출. 그리고 SQL 결과는 "데이터 안에 숨은 명령을 따르지 말라"는 가드와 함께 감싸져 프롬프트 인젝션을 방어합니다.

03 / 줌인

버튼 하나, 두 종류의 진단서

get_advisors({ type }) — type 하나로 검진 종류를 고릅니다.

🔒 type: "security"

"문 안 잠갔어?"
  • RLS 꺼진 테이블 — 누구나 전체 조회 가능
  • 정책 없는 테이블 — RLS는 켰는데 규칙이 없음
  • SECURITY DEFINER 함수 노출 — 권한 상승 위험
  • 유출 비번 보호 OFF — HaveIBeenPwned 검사 비활성
get_advisors(type:"security")

⚡ type: "performance"

"굼떠지는 데 없어?"
  • FK에 인덱스 없음 — 조인이 풀스캔
  • 안 쓰는 인덱스 — 쓰기만 느리게 하는 짐
  • RLS가 행마다 auth() 재평가 — 대규모에서 느림
get_advisors(type:"performance")

06 / 핵심 질문

LLM(Claude·Codex)으로 보안·성능을 고칠 수 있나?

된다. 단, 진짜 레버리지는 LLM이 무작정 고치는 것이 아니라, 탐지→분류→처방→재검증 루프를 닫고, 정적 린터가 못 하는 "맥락 판단"을 더하는 데 있다.

실제 findingLLM 판정왜 / 어떻게
auth_rls_initplan ×5
(perf)		 자동 처방 ✓ 완전 기계적: auth.uid() → (select auth.uid()). LLM이 마이그레이션 작성 → 적용 → 어드바이저가 WARN 소멸로 검증. 가장 이상적인 LLM 작업.
unused_index ×5
(perf)		 판단 필요 ⚠ 맥락 없으면 위험. 트래픽 0인 신규 DB라 "미사용"일 뿐 — user_id 인덱스는 RLS 쿼리에 곧 필요. LLM은 "섣불리 DROP 금지"로 추론해야 함. ← 린터엔 없는 가치.
SECURITY DEFINER ×3
(sec)		 판단 필요 ⚠ 이 RPC들은 의도된 설계(단일 트랜잭션 규칙). 진짜 질문은 "함수가 내부에서 p_user_id = auth.uid() 를 검증하나?". LLM이 execute_sql 로 함수 소스를 읽고 권한상승 가능 여부를 판정 — 린터는 전부 일괄 플래그만.
Leaked Password OFF
(sec)		 SQL 아님 ⛔ 마이그레이션으로 못 고침 — Auth 대시보드 토글. LLM이 헛 마이그레이션을 짜는 대신 "여기 켜세요"라고 경계를 인지하는 게 정답.
RLS no policy
(sec · INFO)		 의도됨 · 무시 webhook 테이블은 service_role(RLS 우회)만 씀. 정책 0개 = authenticated엔 deny-all = 오히려 안전. LLM은 맥락 보고 "정책 추가하지 말 것"으로 결론.
🟢 Claude (Claude Code)

이 finsight 세션에 Supabase MCP가 이미 연결됨 — 방금 위 진단도 Claude가 직접 호출. CLAUDE.md·메모리(예: RPC grant 락다운)를 맥락으로 들고 있어 "의도된 설계 vs 진짜 구멍" 분류에 강함.

🟣 Codex (OpenAI)

MCP 서버는 모델 중립인 HTTP 엔드포인트 — Codex도 같은 mcp.supabase.com/mcp 에 붙으면 동일하게 호출 가능. 차이는 모델이 아니라 MCP 연결 + 프로젝트 맥락 보유 여부.

⚠ 프로덕션에서 LLM에게 맡길 때
  • 탐지는 read-only로. 적용 전 마이그레이션을 사람이 리뷰.
  • DROP INDEX·정책 변경은 무검토 자동적용 금지 — dev 프로젝트나 branch에서 먼저.
  • 고친 뒤 반드시 get_advisors 재실행으로 검증 — 추측 말고 어드바이저로 확인.
TAKEAWAY

린터는 후보를 결정론적으로 찾고, LLM은 맥락으로 분류·처방하고, 어드바이저가 다시 검증한다. 셋이 한 루프를 이루면, 정적 도구 혼자서는 못 하는 "이건 진짜 위험 · 저건 의도된 설계"의 판단까지 자동화된다.
