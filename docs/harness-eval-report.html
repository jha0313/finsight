01

무엇을, 왜 재는가

코드에 테스트가 있다면, 하네스엔 Eval이 있다. 두 축을 본다 — 리뷰 하네스가 룰 위반을 일관되게 잡는가(review), 그리고 라이브 CLAUDE.md가 코드베이스 질문에 정확히 답할 만큼 충분한가(qa). gotcha 한 줄을 CLAUDE.md에서 지우면 해당 qa 케이스가 회귀로 잡힌다.

02

플로우

golden set cases/*.md frontmatter=라벨(정답) · 본문=subject 입력. kind로 트랙을 가른다.
▶
kind=review Reviewer Sonnet · temp 0 · CLAUDE.md CRITICAL 룰 요약을 system prompt로. 코드→산문 리뷰.
kind=qa Responder Sonnet · temp 0 · 라이브 CLAUDE.md를 컨텍스트로. 질문→산문 답변.
▶
llm-as-judge Judge · Opus 다른 모델이 채점. review=expect/severity · qa=must/must_not. structured pass|fail.
▶
회귀 게이트 run.ts kind로 분기·집계. 하나라도 fail이면 exit 1. CI/로컬 회귀 차단.
분리 파서(lib/case.ts)·집계(lib/verdict.ts)는 순수 함수 — 키 없이 npm test가 무결성·균형을 지킨다. 네트워크·비용 있는 라이브 채점만 npm run eval로 분리.
03

두 트랙

① review

subject: Reviewer (Sonnet · temp 0) · judge: Opus
  • 측정 — 리뷰 하네스가 CRITICAL 룰 위반을 잡고, 정상 코드를 오탐하지 않는가.
  • 입력 — 본문=코드(diff/스니펫). 라벨=expect(violation·pass)/severity/rule.
  • 균형 — 위반 4 + 정상(오탐 방지) 1. 정상 케이스가 거짓 양성 회귀를 잡는다.
  • 채점 — judge가 기대 위반 지적 + 심각도 부합을 본다.

② qa

subject: Responder (Sonnet · temp 0 · live CLAUDE.md) · judge: Opus
  • 측정 — CLAUDE.md가 규약·예외처리·gotcha에 답할 만큼 충분한가 + 틀린 전제에 동조 안 하는가.
  • 입력 — 본문=질문. 라벨=must(담겨야 할 사실)/must_not(오답).
  • 균형 — 사실 검증 3 + 전제반박 가드 1(환각 방지).
  • 센티넬 — positive 케이스는 추론 테스트가 아니라 CLAUDE.md 커버리지 회귀 감지기(그 줄 지우면 red).
04

Golden set — 9 케이스

라벨이 정답이고 본문이 subject에게 줄 입력이다. 라벨은 사람이 검수해 박제한다 — 자동 라벨은 judge와 같은 모델 편향을 공유해 회귀를 놓친다.

REVIEW · 위반 4 + 정상 1
01-lib-imports-sdk reviewviolationcritical
RULE 레이어 의존성은 단방향. lib/는 외부 SDK(@anthropic-ai/sdk 등)를 import하지 않고, 어댑터는 route handler에서 주입한다.
// src/lib/analysis/summarize.ts
+import Anthropic from "@anthropic-ai/sdk";
+  const client = new Anthropic({ apiKey: process.env... });
02-money-parsefloat reviewviolationcritical
RULE 금액은 numeric으로 다루고 float(parseFloat 등)을 쓰지 않는다. 통화기호·콤마·괄호음수를 정규화한다.
// src/lib/csv/amount.ts
+  return parseFloat(raw.replace(/[$,]/g, ""));
03-service-role-public-env reviewviolationcritical
RULE service_role 키는 웹훅 모듈에서만 쓰고 server-only로 가드. NEXT_PUBLIC_에 비밀키 금지(클라이언트 번들 노출).
// src/lib/supabase/admin-client.ts
+  process.env.NEXT_PUBLIC_SUPABASE_SERVICE_ROLE_KEY!,
04-pro-gating-request-body reviewviolationcritical
RULE Pro 게이팅은 서버측 DB 구독상태(status active AND current_period_end > now())로만 판정. 요청 본문/헤더의 tier를 신뢰하지 않는다.
// src/app/api/analyze/route.ts
+  const body = await request.json();
+  const isPro = body.tier === "pro";
05-lazy-provider-injection reviewpass
오탐 방지 포트 인터페이스(types/)에만 의존하고 주입된 factory로 호출 시점에 지연 생성하는 정상 코드. 외부 결합을 다루는 듯 보여도 SDK import가 없으므로 위반이 아니다.
// src/lib/orchestration/insight-provider.ts
import type { InsightProvider } from "@/types/ports";
  provider ??= insightProviderFactory();
QA · 사실 검증 3 + 전제반박 가드 1
q01-pro-gating-unsubscribed-status qa
“미구독 사용자가 /api/analyze를 호출하면 402로 막으면 되나요?”
must
  • 402가 아니라 HTTP 200으로 응답
  • Free(규칙·통계) 결과를 함께 반환
  • pro.status를 locked/unavailable로 표시
must_not
  • 402 Payment Required 반환
  • 결제 전까지 분석 거부
q02-claude-timeout-isolation qa
“Pro 심층 분석 중 Opus 호출이 타임아웃됩니다. 요청 전체를 500으로 실패시키면 되나요?”
must
  • 요청 전체를 실패시키지 않는다
  • 규칙·통계 결과는 보존
  • AI 인사이트만 unavailable로 격리
  • Opus 호출에 timeout(예 30s)
must_not
  • 요청 전체를 500으로 실패
  • 규칙·통계 결과까지 버림
q03-csv-summary-rows-gotcha qa
“CSV에 합계·소계 요약행이 섞이고 금액에 통화기호·콤마·괄호음수가 있습니다. 파싱에서 무엇을 주의해야 하나요?”
must
  • 합계·소계 요약행을 필터링
  • 통화기호·콤마·괄호음수 정규화
  • float이 아니라 numeric으로
  • 부호 규약을 direction(debit/credit/refund)으로 단일화
must_not
  • parseFloat로 금액 파싱
  • 요약행을 거래처럼 합산
q04-sync-not-async-premise qa전제반박 가드
틀린 전제 ▸ “finsight는 분석을 백그라운드 작업 큐(비동기 job)로 처리하죠? 그 큐 이름이 뭔가요?”
must
  • 비동기 큐가 아니라 동기 처리다(전제 바로잡기)
  • 저장은 RPC 단일 트랜잭션(save_statement_analysis)
  • 중간 실패 시 전체 rollback
must_not
  • 비동기 큐가 맞다고 동조
  • 존재하지 않는 큐 이름을 지어냄(환각)

06

왜 두 명령으로 나뉘나

npm test

키 불필요 · 결정적 · 빠름. 순수 함수와 golden set 무결성만 검증하는 green 게이트.

  • parseCase — frontmatter/본문 분리, 잘못된 라벨 throw
  • loadCases — 균형 박제: review 위반4+정상1, qa must>0·가드≥1, id 유일
  • summarize/formatSummary — 집계·리포트
  • system prompt — 룰 전량 포함·전제반박 지시
npm run eval

네트워크·비용·비결정. 라이브 Claude 호출로 subject를 돌리고 Opus가 채점. .env 자동 로드, 키 없으면 exit 2.

  • review 5 → Reviewer(Sonnet) → Judge(Opus)
  • qa 4 → Responder(Sonnet) → Judge(Opus)
  • 하나라도 fail → exit 1 (회귀 게이트)
  • 일시 과부하(529) 대비 maxRetries:5 백오프

한계 — 의도된 트레이드오프

!
judge는 흔들린다. 리뷰어는 temp 0으로 고정하지만 Opus judge는 temperature를 지원하지 않아 채점이 완벽히 결정적이지 않다. 애매하면 사람이 최종 검토(judge 맹신 금지).
≈
경량 리뷰어 ≠ 실제 /review-code. 게이트 속도를 위해 단일 호출로 대리한다. 산문 in/out 계약이라 reviewer.ts만 headless /review-code로 교체하면 실제 워크플로우를 eval할 수 있다(judge·게이트 불변).
∋
qa 응답자는 CLAUDE.md를 통째로 받는다. 사실이 거기 있으면 거의 맞힌다 — positive qa는 추론 테스트가 아니라 커버리지 회귀 센티넬이다. 진짜 변별력은 전제반박 가드와, 아직 CLAUDE.md에 없는 사실을 묻는 케이스에서 나온다.
