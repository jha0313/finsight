Claude Code · 멀티에이전트 오케스트레이션

여러 에이전트를
결정론적으로 지휘하는 도구

Workflow는 JavaScript 스크립트 한 편으로 수십 개의 서브에이전트를 병렬·파이프라인으로 돌리는 도구입니다. 제어 흐름(루프·분기·팬아웃)은 코드가 결정론적으로 잡고, 각 단계의 판단만 에이전트에게 맡깁니다. 실행은 백그라운드에서 진행되고 /workflows로 실시간 진행 상황을 봅니다.

왜 쓰는가

01한 컨텍스트로는 벅찬 일을 나눠 맡길 때

혼자 처리하면 느리거나, 빠뜨리거나, 확신이 안 서는 작업이 있습니다. Workflow는 그 일을 구조로 풉니다. 쓰는 이유는 보통 셋 중 하나입니다.

철철저함

작업을 쪼개 병렬로 커버. 큰 코드베이스 감사, 광범위한 검색, 전수 점검처럼 빠뜨림이 치명적인 일에.

확확신

독립적인 관점과 adversarial 검증을 거쳐 결론을 굳힘. 그럴듯하지만 틀린 결과가 살아남지 못하게.

규규모

한 컨텍스트에 안 들어가는 규모. 마이그레이션, 대규모 리팩터, 코드베이스 전체 sweep.

전제 조건

02반드시 사용자가 명시적으로 켜야 한다

Workflow는 에이전트를 수십 개 띄우고 토큰을 많이 씁니다. 그래서 옵트인이 강제됩니다 — "이득이 될 것 같아서" 추론으로 켜는 일은 없습니다.

트리거
다음 중 하나일 때만 실행됩니다 — 프롬프트에 "ultracode" 키워드가 있을 때, 세션에 ultracode가 켜져 있을 때, 사용자가 자기 말로 "워크플로우 돌려줘 · 에이전트 팬아웃해줘 · 멀티에이전트로 처리해줘"라고 직접 요청할 때, 또는 그렇게 지시하는 스킬/슬래시 명령을 부를 때. 그 외에는 — 명백히 이득인 작업이라도 — 단일 Agent를 쓰거나 비용을 먼저 알리고 물어봅니다.
스크립트 구조

03meta 선언 + 본문, 단 두 부분

모든 스크립트는 순수 리터럴 meta 객체로 시작합니다(변수·함수호출·스프레드 금지). 그 아래 본문은 agent() / parallel() / pipeline() / phase() / log()로 짭니다.

workflow.js — 기본 골격
export const meta = {
  name: 'find-flaky-tests',
  description: 'Find flaky tests and propose fixes',  // 권한 다이얼로그에 표시
  phases: [                                          // phase() 호출마다 한 줄
    { title: 'Scan', detail: 'grep test logs for retries' },
    { title: 'Fix',  detail: 'one agent per flaky test' },
  ],
}
// ── 여기서부터 본문 (async 컨텍스트 — await 직접 사용) ──
phase('Scan')
const flaky = await agent('grep CI logs for retry markers', { schema: FLAKY_SCHEMA })

phase('Fix')
const fixes = await pipeline(flaky.tests,
  t => agent(`Reproduce flake: ${t.name}`, { schema: REPRO }),
  r => agent(`Propose a fix for: ${r.cause}`, { schema: FIX }),
)
return { fixes }
주의
스크립트는 TypeScript가 아니라 순수 JS입니다 — 타입 주석·인터페이스·제네릭은 파싱 실패. 그리고 Date.now() · Math.random() · 인자 없는 new Date()는 throw합니다(resume를 깨뜨리므로). 타임스탬프는 args로 주입하거나 워크플로우가 끝난 뒤 찍으세요.
빌딩 블록

04다섯 개의 훅

훅역할
agent(prompt, opts?) 서브에이전트 하나를 띄움. schema를 주면 검증된 객체를, 안 주면 최종 텍스트를 반환. label·phase·model·effort·isolation:'worktree'·agentType 지정 가능.
pipeline(items, …stages) 기본 선택지. 각 item을 모든 stage에 독립적으로 통과시킴 — 단계 사이 배리어 없음. item A가 stage 3일 때 item B는 아직 stage 1일 수 있음.
parallel(thunks) 배리어. 모든 thunk를 동시에 돌리고 전부 끝날 때까지 기다림. 결과를 한꺼번에 모아야 할 때만.
phase(title) · log(msg) 진행 표시를 그룹핑 / 내레이터 줄로 진행 메시지 출력.
args · budget · workflow() 입력값 주입 / 토큰 예산(budget.remaining()로 동적 루프) / 다른 워크플로우를 한 단계로 인라인 실행(1단계 중첩까지).
가장 중요한 선택

05pipeline 먼저, parallel은 정말 필요할 때만

이게 Workflow 설계의 핵심입니다. 배리어(parallel)는 벽시계 시간을 낭비합니다 — 빠른 에이전트가 느린 에이전트를 기다리며 놀게 되니까요. stage N이 stage N-1의 전체 결과를 교차 참조해야 할 때만 배리어가 정당합니다.

parallel 결과를 그냥 flatten/map/filter한 뒤 다시 parallel에 넣고 있다면 — 그 중간 변환엔 배리어가 필요 없습니다. pipeline stage 안에서 변환하세요. "코드가 더 깔끔해서"는 배리어의 이유가 아닙니다.
품질 패턴

06구조로 정확도를 끌어올린다

단순 팬아웃을 넘어, 결과의 신뢰도를 높이는 검증된 형태들. 작업에 맞게 골라 조합합니다.

검

Adversarial verify

발견마다 독립적인 회의론자 N명을 띄워 반증하게 함. 과반이 반증하면 폐기. 그럴듯하지만 틀린 결과를 걸러냄.

렌

Perspective-diverse verify

실패 방식이 여러 가지면 검증자마다 다른 렌즈(정확성·보안·성능·재현성)를 부여. 중복보다 다양성이 더 많은 실패를 잡음.

심

Judge panel

서로 다른 각도(MVP우선·리스크우선·사용자우선)로 N개 안을 생성 → 병렬 심사로 점수 → 승자 기반으로 종합하되 차점안의 좋은 아이디어를 이식.

건

Loop-until-dry

크기를 모르는 발견(버그·엣지케이스)은 K라운드 연속으로 새 결과가 없을 때까지 finder를 계속 띄움. 단순 카운터는 꼬리를 놓침.

탐

Multi-modal sweep

병렬 에이전트가 각자 다른 방식(컨테이너별·내용별·엔티티별·시간별)으로 검색. 한 검색 각도로는 다 못 찾을 때.

완

Completeness critic

마지막에 "무엇이 빠졌나 — 안 돌린 모달리티, 미검증 주장, 안 읽은 출처는?"을 묻는 에이전트. 그 답이 다음 라운드 작업이 됨.

제약과 한계

07알아둘 경계선

∥동시성 캡

동시 실행은 min(16, 코어수−2)개. 초과분은 큐에 쌓여 슬롯이 비면 실행. 그래서 100개를 넘겨도 다 끝나긴 함.

∞총량 캡

워크플로우 수명 전체 에이전트 1000개 상한(런어웨이 루프 방지용 backstop). 한 parallel/pipeline 호출은 최대 4096 item.

$토큰 budget

budget.total이 하드 천장. spent()가 도달하면 이후 agent()는 throw. 메인루프+모든 워크플로우가 풀을 공유.

↻Resume

scriptPath+resumeFromRunId로 재개. 안 바뀐 agent() 호출은 캐시 결과 즉시 반환, 수정된 첫 호출부터 라이브 실행.

/workflows
Workflow는 백그라운드에서 돌고, 호출 즉시 task ID를 반환합니다. 진행 트리를 실시간으로 보려면 /workflows를 쓰고, 완료되면 <task-notification>이 도착합니다. 반복하려면 매번 풀 스크립트를 다시 보내지 말고, 반환된 scriptPath 파일을 수정해 재호출하세요.
정석 예시

08차원별 리뷰 → 발견마다 즉시 검증

pipeline의 진가 — 'bugs' 차원의 발견이 검증되는 동안 'perf' 차원은 아직 리뷰 중. 낭비되는 벽시계 시간이 없습니다.

review-changes.js
const DIMENSIONS = [{ key: 'bugs', prompt: '…' }, { key: 'perf', prompt: '…' }]

const results = await pipeline(
  DIMENSIONS,
  d => agent(d.prompt, { label: `review:${d.key}`, phase: 'Review', schema: FINDINGS }),
  review => parallel(review.findings.map(f => () =>
    agent(`Adversarially verify: ${f.title}`, { label: `verify:${f.file}`, phase: 'Verify', schema: VERDICT })
      .then(v => ({ ...f, verdict: v }))
  ))
)

const confirmed = results.flat().filter(Boolean).filter(f => f.verdict?.isReal)
return { confirmed }
