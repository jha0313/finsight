Supabase Branching · deployment/branching

실험은 복제본에서
Staging = Supabase Branching

권고를 받았으면 고쳐서 테스트해야죠. 그런데 어디서? 프로덕션에서 마이그레이션을 바로 돌리는 건 절대 금물. Branching은 Git 브랜치처럼 격리된 DB 인스턴스를 통째로 복제해서, 진짜 같은 환경에서 안전하게 실험하게 해줍니다.

01 / 핵심 개념

Branch = 격리된 Supabase 인스턴스 통째로

브랜치는 스키마만 복제하는 게 아니라 DB·Auth·Storage·Realtime·Edge Functions까지 독립된 별도 Supabase 프로젝트입니다. 각자 전용 자격증명/접속 문자열을 받아 프로덕션과 절대 섞이지 않습니다.

🌳
main · Production
라이브 시스템 — 모든 merge가 최종 도착하는 곳
PROD
🔀 feat/fix-rls-initplan PREVIEW
own DB · own keys
postgres://…branch-a.supabase.co
🔀 staging PERSISTENT
own DB · own keys
postgres://…staging.supabase.co
🧬 독립 인스턴스
프로덕션과 물리적으로 분리된 별도 DB
🔑 전용 자격증명
브랜치마다 다른 키·접속 문자열
🛡️ 폭발 반경 0
실험이 망가져도 프로덕션 무사
02 / DB는 어떻게 생기나

"복사"가 아니라 "다시 짓기"

브랜치 DB는 프로덕션 DB 파일을 복사한 스냅샷이 아닙니다. 텅 빈 새 Postgres를 띄운 뒤, 당신의 마이그레이션 SQL을 처음부터 다시 실행해 똑같은 구조를 새로 짓습니다.

❌ 흔한 오해 — 복제완성된 요리(데이터까지)를 통째로 퍼온다. 이게 아닙니다.
✅ 실제 — 재생(replay)레시피(마이그레이션)를 새 주방에서 다시 요리한다. 구조는 같고 재료(데이터)는 새것.
①⬛빈 Postgres독립 인스턴스 새로 프로비저닝
▶
② Migrate · 순서대로 재생
supabase/migrations/*
0001_init.sql 테이블·RLS·함수
0002_..quota.sql 변경 누적
0003_revoke.sql ↓
0004_..fix.sql ↓
0005_..cancel.sql 완성
▶
③🧬쌍둥이 스키마프로덕션과 1:1 동일 구조
▶
④ Seed🌱합성 데이터seed.sql만 — 프로덕션 데이터 0
↳그래서 migrations-as-code가 필수다. finsight의 0001~0005.sql이 곧 "DB를 처음부터 만드는 결정론적 레시피" — 이게 있으니 브랜치가 가능하고, 복사 단계 자체가 없으니 프로덕션 데이터가 새지 않는 게 자동 보장된다.
03 / Git 통합

PR 생명주기에 브랜치가 따라붙는다

GitHub 연동을 켜면, PR을 열고·푸시하고·머지하고·닫는 행위가 그대로 브랜치 환경의 생성·재배포·반영·삭제로 이어집니다. 사람이 손댈 게 없습니다.

PR open
feature 브랜치로 PR 생성
🌱 Preview 브랜치 자동 생성
PR이 열리는 순간 격리된 미리보기 환경이 프로비저닝됨
git push
커밋 추가
🔁 매 커밋마다 재배포
7단계 워크플로가 다시 돌아 브랜치 DB에 마이그레이션·seed·함수 반영
review / test
미리보기 환경에서 검증
🧪 브랜치 DB에서 안전하게 마이그레이션 테스트
프로덕션과 같은 스키마, 다른 데이터 — 망가뜨려도 OK
PR merge
main으로 병합
🚀 프로덕션에 동일 워크플로 자동 실행
머지하면 main 프로젝트에 마이그레이션이 그대로 적용됨 — 검증된 그 변경 그대로
PR close / merged
정리
🗑️ Preview 브랜치 자동 삭제
리소스 회수 — 쓰지 않는 인프라가 쌓이지 않음 (persistent 브랜치는 삭제 안 됨)
04 / 배포 워크플로

커밋마다 도는 7단계

브랜치든 프로덕션이든 동일한 7단계가 순서대로 실행됩니다. 단계가 실패하면 그 아래 의존 단계는 건너뜁니다. 핵심은 Migrate(마이그레이션)와 Seed(시드 데이터).

1
📥
Clone
해당 git 브랜치 체크아웃
2
⬇️
Pull
main의 마이그레이션 가져오기
3
❤️
Health
서비스 기동 확인 (2분)
4
⚙️
Configure
config.toml 적용
5
🧱
Migrate
마이그레이션·vault 적용
6
🌱
Seed
시드 파일로 데이터 주입
7
🚀
Deploy
Edge Functions·secrets
↳같은 워크플로가 머지 시 프로덕션에도 그대로 실행 — 그래서 "브랜치에서 됐으면 프로덕션에서도 된다"가 성립한다.
05 / 가장 중요한 안전장치

스키마는 따라오고, 데이터는 안 따라온다

새 브랜치는 프로덕션 데이터를 단 한 줄도 복제하지 않습니다. 구조(마이그레이션)는 전파되지만, 민감 데이터는 격리된 채 남고 브랜치는 seed 파일로만 채웁니다 — 핀테크/PII 관점에서 결정적입니다.

🧱 따라온다

SCHEMA · 구조

마이그레이션으로 정의된 모든 구조: 테이블, RLS 정책, 함수, 인덱스, 트리거. Pull 단계가 main의 마이그레이션을 가져와 그대로 재현.

supabase/migrations/*.sql → 브랜치 DB
⇄

🔒 안 따라온다

DATA · 실제 행

프로덕션의 실제 거래·카드·계좌·사용자 데이터는 절대 복제 안 됨. 테스트가 필요하면 seed 파일로 가짜/합성 데이터를 넣는다.

supabase/seed.sql → 합성 데이터만
06 / 두 종류의 브랜치

Preview(일회용) vs Persistent(상주)

"Staging"은 보통 Persistent 브랜치로 만들고, 개별 기능 검증은 Preview 브랜치로 합니다.

축🔀 Preview📌 Persistent
목적PR 단위 일회성 검증Staging · QA · 개발 상주 환경
수명PR 열릴 때 생성 → 닫히면 자동 삭제무기한 유지 — PR 머지돼도 안 사라짐
유휴 시비활성 후 자동 일시정지일시정지 안 됨
데이터seed로만 채움 (가벼움)seed 권장 — 누적 테스트 상태 유지 가능
언제"이 PR 한 번 검증하고 버린다""항상 떠 있는 staging이 필요하다"
