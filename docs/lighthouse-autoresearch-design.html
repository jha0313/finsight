keep-or-revert 루프 설계

finsight Lighthouse 최적화를 위한 자율 keep-or-revert 루프의 구조와 동작. 한 반복은 “수정 → 게이트 → 측정 → 판정”이며, 판정 결과가 다음 반복의 기준선(baseline)을 결정한다.

루프 다이어그램

한 반복의 흐름과 피드백

파이프라인은 좌→우로 흐르고, 판정(⑤) 이후 채택/되돌림이 갈린 뒤 보라색 피드백 경로를 따라 다음 반복으로 돌아간다. 게이트(③) 실패나 회귀(⑤)는 곧바로 REVERT로 떨어진다.

파이프라인 단계 채택(개선 & green) 되돌림(게이트 실패·회귀) 종료 조건 피드백 경로

한 반복의 해부

SELECT → EDIT → VERIFY → MEASURE → DECIDE

1

SELECT

PROGRAM.md 백로그에서 ROI 최상위 1개. 가설·파일·예상 Δ 명시.

2

EDIT

외과적 구현. 디자인 토큰만, hex 금지. 변경 줄은 그 항목과 직접 연결.

3

VERIFY

lint && build && test green이어야 통과. red면 즉시 REVERT.

4

MEASURE

로컬 prod 빌드에 LH 2×2(페이지×기기), 타깃별 median-of-3.

5

DECIDE

목적함수 ↑ & 무회귀면 commit·baseline 갱신, 아니면 폐기. 종료 조건 확인.

하베스트 구성

측정 하네스 · 파일 레이아웃

루프의 결정론은 측정 하네스에서 나온다. 앱 코드와 분리된 scripts/lh/ 아래에 측정·우회·판정·기준선을 둔다.

scripts/lh/측정 하네스
scripts/lh/
├─ measure.mjs     # build → start(격리 .next-lh/포트) → LH 매트릭스 → scores.json
├─ run.sh          # F-1 미들웨어 우회 임시 적용 → measure → 원복(/dashboard 측정용)
├─ objective.mjs   # 가중 점수 + 회귀 가드(ε=1pt) → keep/revert 판정
├─ PROGRAM.md      # 제약·목적·ROI 백로그 — 사람이 편집해 방향 제어
├─ baseline.json   # 현재 채택된 점수 기준선 (keep마다 갱신)
└─ reports/        # 반복별 LH HTML 리포트 보관 (감사 추적)

병렬 vs 순차

진단은 팬아웃, 루프는 순차

진단(Phase 1)은 카테고리별로 독립적이라 병렬 Workflow가 맞다. 최적화 루프(Phase 2)는 매 반복이 직전 코드 상태에 의존하는 상태 누적이라 병렬화할 수 없다.

Phase 1 · 진단 /workflows 팬아웃

LH 감사 결과를 입력으로 카테고리별 finder 에이전트를 병렬 실행해 ROI 백로그를 만든다.

  • 카테고리(Perf·A11y·BP·SEO)별 동시 분석
  • 각 발견 = {가설, 파일, 예상 Δ, 리스크, 적합성}
  • dedup → ROI 랭킹 → PROGRAM.md
Phase 2 · 최적화 순차 구동

keep-or-revert는 baseline이 매 반복 갱신되는 순차 상태기계라 Workflow에 부적합하다.

  • 반복 N의 시작 = 반복 N−1이 채택한 코드
  • 측정·커밋·되돌림이 단일 작업트리에서 순서대로
  • 충돌·경합 방지 위해 동시 변경 없음

사람의 제어점

PROGRAM.md

human-in-the-loop

탐색은 자율, 가드레일은 사람이 정의

autoresearch에서 연구자가 program.md만 편집하듯, 여기서는 scripts/lh/PROGRAM.md에 제약(디자인 토큰만 · 외과적 수정 · 인증/RLS/Pro 게이팅 불변 · 측정 픽스처 커밋 금지), 목적함수 정의, ROI 백로그를 둔다. 방향을 바꾸려면 이 문서만 수정하고 루프 코드는 그대로다. 점수를 올리려 제품을 망치는 해킹성 변경은 회귀 가드와 green 게이트가 자동으로 막는다.
