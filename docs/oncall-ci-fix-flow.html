FinSight · Oncall 운영 에이전트

CI 실패 → 자동 분석 → 수정 PR

사람이 시킨 게 아니라 시스템이 깨진 사실 자체가 트리거다(사고 대응). 기본 CI(lint·build·test)가 실패로 끝나면, 헤드리스 Claude 에이전트가 실패 잡 로그만 읽어 근본 원인을 짚고 수정 브랜치 → PR을 연다. 출력은 무조건 PR(자동 머지 금지) — 최종 머지는 사람이 한다.

0 · 트리거 소스

개발자 변경 push → main · 또는 PR open/sync

커밋이 올라오면 기본 CI가 자동으로 돈다. 여기엔 secret이 필요 없다 — 외부 클라이언트는 호출 시점 지연 생성이라 키 없이 build/test가 통과해야 정상.

1 · 기본 CI .github/workflows/ci.yml

lint · build · test node 22 · npm ci

ESLint(--max-warnings=0) → next build(타입체크 포함) → vitest run. 깨질 대상이자, 자동수정 루프의 진입 신호.

결과 판정
✅ conclusion = success ❌ conclusion = failure
↳ 통과하면
종료 — 아무 일도 하지 않음. oncall은 깨어나지 않는다.
실패일 때만 ↓
2 · 이벤트

workflow_run · workflows:["ci"] · completed

CI가 실패로 완료되면 후속 워크플로가 깨어난다. workflow_run은 기본 브랜치(main)에 있는 워크플로 정의만 추적한다 — 그래서 ci.yml·oncall-ci-fix.yml이 main에 있어야 동작한다.

3 · GUARD (if:) .github/workflows/oncall-ci-fix.yml

실행 전 4중 가드 — 무한루프·포크·봇 차단

  • ✓
    conclusion == 'failure' — 실패한 run만
  • ✓
    !startsWith(head_branch, 'oncall/') — 에이전트가 만든 브랜치의 실패는 무시(자기 자신을 다시 깨우지 않음 = 무한루프 1순위 가드)
  • ✓
    head_repository == repository — 포크 PR 차단(신뢰 못 할 코드에 secret 노출 금지)
  • ✓
    actor != 'github-actions[bot]' — 봇이 트리거한 실패 차단
↳ 하나라도 불충족이면
SKIP — 잡이 아예 실행되지 않음. 봇/oncall 브랜치/포크가 만든 실패는 여기서 조용히 끊겨 무한루프를 막는다.
모두 충족 ↓
4 · 잡 셋업

실패 브랜치 체크아웃 · node 22 · npm ci

실패한 브랜치를 fetch-depth: 0으로 받아 main 대비 diff를 본다. 에이전트가 수정을 직접 lint/build/test로 검증할 수 있도록 의존성을 설치한다.

5 · 헤드리스 에이전트

anthropics/claude-code-action@v1 하네스: SKILL.md · ci-fix.md · service-map.md

인터랙티브 터미널과 동일한 oncall 하네스를 읽고 따른다. 허용 도구는 Read/Edit/Write · git · gh run view · gh pr create/list · npm run *로 좁혀져 있다.

  1. 실패 로그 수집 — gh run view <id> --log-failed · 실패 잡 로그만 읽는다(성공 잡은 노이즈).
  2. 근본 원인 분석 — 타입 에러·누락 import·테스트 미스매치 등. CLAUDE.md CRITICAL 규칙 위반 수정 금지.
  3. 수정 브랜치 — oncall/fix-<run_id> 를 새로 딴다. 원본/main에 직접 push·머지 금지.
  4. 패치 + 검증 — 수정 후 npm run lint/build/test 로 green 확인.
  5. PR 생성 — git push → gh pr create --base <실패 브랜치>.
6 · 출력 = PR

수정 PR base = 실패 브랜치 · head = oncall/fix-<run_id>

본문엔 무엇이 왜 깨졌고(분석) 어떻게 고쳤는지(패치 요약) + 검증 결과를 담는다. 로그 원문이 아니라 분석 요약만 — 키/토큰 같은 secret 패턴은 본문에서 제거. 같은 run엔 PR 하나(중복 생성 금지, 기존 브랜치 갱신).

🚫 자동 머지·approve 절대 금지 — 사람 게이트
7 · 루프 닫힘

code-review.yml 자동 리뷰 → 👤 사람이 머지

이 PR은 기존 리뷰 워크플로가 자동으로 본다. 에이전트는 "고쳤다"까지, "반영했다"는 사람이 결정한다. 이 PR의 CI가 또 돌아도 head_branch가 oncall/*라 가드(3)에서 skip — 루프가 다시 안 돈다.

항상 지키는 게이트

🛑
자동 머지 금지
출력은 PR. 고침과 반영 사이에 사람을 한 명 세운다.
🔁

실패 잡 로그만 본다

실패 스텝 로그 + 변경 diff → 컨텍스트로 사용
성공 잡 로그 → 보지 않음(노이즈)
