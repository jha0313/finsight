# UI/UX 가이드: 획득과 단계 전환 설계

이 문서는 finsight의 화면을 **유저 획득(acquisition)과 단계별 전환(activation funnel)** 관점에서 설계·검수하기 위한 기준이다. 시각 토큰·컴포넌트 규칙의 단일 출처는 [`docs/DESIGN.md`](./DESIGN.md)와 `.claude/skills/vantage-design/`이며, 이 문서는 그 위에서 **"무엇을, 왜, 어느 순서로 보여줄 것인가"**를 다룬다.

원칙의 출처는 [Figma — UI Design Principles](https://www.figma.com/resource-library/ui-design-principles/)의 7가지 원칙(Hierarchy / Progressive Disclosure / Consistency / Contrast / Accessibility / Proximity / Alignment)이며, 아래에서 이를 finsight 맥락으로 번역해 적용한다.

---

## 0. 한 줄 원칙

> **모든 화면에는 "지금 해야 할 단 하나의 행동"이 있고, 그 행동은 화면에서 가장 강한 시각 신호를 받는다.**

finsight는 이 원칙을 시스템 차원에서 강제한다. Vantage의 **단일 accent(`--primary` #0052ff는 CTA·링크에만)** 규칙 때문에, 화면에서 파란색을 띤 요소는 곧 "다음 단계"다. 디자인 제약이 곧 전환 설계다 — 한 화면에 파란 버튼이 두 개 이상 경쟁하면 그것은 디자인 위반인 동시에 퍼널 누수다.

---

## 1. Figma 7원칙 → finsight 번역

각 원칙을 추상론이 아니라 finsight의 구체 화면 규칙으로 환산한다.

### 1) Hierarchy (위계) — "먼저 보이는 것"을 의도적으로 정한다
- 모든 화면은 **단 하나의 1차 행동**을 가진다. 그것만 accent 채움 버튼(pill). 나머지는 텍스트 링크 또는 hairline 보더 버튼으로 강등한다.
- 데이터 화면(대시보드)의 위계 순서: **결론 숫자(이번 달 지출·전월 대비) → 구조(카테고리·추이) → 이상치 → AI 인사이트**. 사용자가 스크롤 없이 "결론"을 먼저 본다.
- display는 weight 400을 유지(DESIGN.md). 위계는 **굵기가 아니라 크기·여백·색 대비**로 만든다.

### 2) Progressive Disclosure (점진적 공개) — finsight의 핵심 무기
- finsight의 가치는 "임의의 CSV → 이해 가능한 화면"이다. 한 번에 모든 걸 보여주면 압도된다. **한 화면 = 한 결정**으로 쪼갠다.
- 업로드는 `파일 선택 → (필요 시) 컬럼 매핑 1회 확인 → 분석`의 단계로만 노출한다. 표준 파서가 성공하면 매핑 단계는 **아예 보이지 않는다**(LLM 0 경로 = 마찰 0 경로).
- Pro의 깊이는 처음부터 다 보여주지 않는다. Free 결과를 먼저 완성도 있게 주고, 그다음 잠긴 한 칸을 **미리보기(blur/티저)**로 드러낸다.

### 3) Consistency (일관성) — 학습 비용을 0으로
- 1차 행동 버튼은 어느 화면에서나 같은 모양(accent pill)·같은 위치(섹션 하단 또는 카드 우측). 사용자가 "다음으로 가려면 파란 pill"을 한 번 학습하면 끝까지 통한다.
- 금액·%·날짜는 **전부 JetBrains Mono(tabular)**. 한 화면에서 숫자 서체가 섞이면 신뢰가 깨진다.
- 부호→색 매핑은 전역 고정: 지출/이상 = `--semantic-down`, 환불/절약 = `--semantic-up`. 색은 **텍스트에만**, 배경 fill 금지.

### 4) Contrast (대비) — 긴급도에 비례한 강조
- 1차 CTA만 고대비(accent 채움). 위험·되돌릴 수 없는 행동(삭제 등)은 톤만 다르게, accent를 쓰지 않는다(파란색 = 전진의 의미를 오염시키지 않는다).
- 80%는 flat + 1px hairline(#dee1e6), 그림자는 단일 티어 hover에만(DESIGN.md). **대비를 남발하면 1차 행동이 묻힌다.**
- 이상 거래·절약 수치는 색 대비로 "여기 봐라"를 만들되, 트레이딩 그린/레드는 텍스트 색으로만.

### 5) Accessibility (접근성) — 한 화면도 예외 없이
- 모든 인터랙티브 요소는 키보드 도달·포커스 링·`aria-label`. 아이콘 버튼은 텍스트 라벨 동반(현재 `aria-hidden`으로 장식 아이콘 처리 중 — 유지).
- 색만으로 의미를 전달하지 않는다. 상승/하락은 색 + 기호(▲▼ 또는 +/−) + mono 숫자를 함께.
- 명도 대비 WCAG AA 이상. accent(#0052ff) on white, on-dark 텍스트는 대비 확인(메모리: on-dark 텍스트는 `!important` 필요 케이스 있음).
- 로딩/에러/빈 상태에 텍스트 안내 — 스피너만 두지 않는다.

### 6) Proximity (근접성) — 관계있는 것끼리 묶는다
- 한 카드 = 한 개념. 카테고리 도넛과 그 범례·합계는 같은 카드 안. 24px radius 카드 + 32px 패딩이 묶음의 경계다.
- CTA는 그것이 가리키는 가치 진술 **바로 아래** 둔다(가격 설명 → 그 plan의 버튼). 멀리 떨어진 버튼은 클릭률이 떨어진다.
- 96px 섹션 리듬으로 서로 다른 주제(가치 / 보안 / 가격)는 충분히 떨어뜨린다.

### 7) Alignment (정렬) — 예측 가능성 = 신뢰
- 단일 그리드. 숫자는 우측 정렬(tabular-nums가 자리수를 맞춤), 라벨은 좌측 정렬.
- 카드·CTA·헤드라인이 같은 좌측 기준선에 맞으면 "정돈된 금융 제품"으로 읽힌다. 금융 제품에서 어긋난 정렬은 곧 "허술함 = 불신"이다.

## 2. 퍼널 단계별 설계 (문서의 핵심)

각 단계마다 **(a) 사용자 심리 (b) 이 화면의 단 하나의 목표 (c) 적용 원칙 (d) 다음 단계로 넘기는 트리거 (e) 마찰 제거 (f) 검수 체크**를 정의한다.

퍼널: **랜딩 → 가입 → 첫 업로드(Aha) → Free 결과 → Pro 미리보기 → 결제 → Pro 활성 → 재방문**

```
[랜딩]──샘플 데모로 가치 증명──▶[Google 로그인]──1클릭──▶[업로드]
   ▲                                                        │
   │                                                   첫 인사이트(Aha)
[재방문]◀──월 주기 알림──[Pro 활성]◀──결제──[Pro 미리보기]◀──[Free 결과]
```

---

### 단계 1 — 랜딩 (첫 방문자: "이거 믿을 만한가? 나한테 쓸모 있나?")
- **단 하나의 목표:** "Google로 시작" 클릭.
- **전환 트리거:** *증명 먼저, 가입 나중.* Hero 아래 **샘플 데모 섹션**(`SampleDemoSection`)에서 가입 없이 실제 분석 결과를 보여준다 — 이것이 finsight의 가장 강한 획득 장치다. 말로 설명하지 말고 결과물을 보여준다.
- **적용 원칙:** Hierarchy(헤드라인 → 가치 한 줄 → 단일 CTA), Progressive Disclosure(가치→보안→가격 순으로 의심을 차례로 해소), Proximity(가격 설명 바로 아래 plan CTA).
- **신뢰 설계:** `SecuritySection`을 가격 **앞**에 둔다. 금융 데이터 업로드의 가장 큰 마찰은 "안전한가?"이므로, 결제를 고민하기 전에 보안 경계(마스킹·RLS·마스킹 전송)를 먼저 해소한다.
- **마찰 제거:** 가입 전에는 어떤 개인정보도 요구하지 않는다. 데모는 정적이라도 즉시 렌더(`getSampleDemoAnalysis` 서버 프리렌더).
- **검수 체크:**
  - [ ] 첫 화면(above the fold)에 파란 pill CTA가 **정확히 하나**.
  - [ ] 스크롤 없이 "무엇을 해주는 제품인지" 한 문장으로 읽힌다.
  - [ ] 가격보다 보안 섹션이 먼저 온다.
  - [ ] 'Vantage' 문자열 노출 0, 워드마크 = finsight.

### 단계 2 — 가입 (Google OAuth)
- **단 하나의 목표:** 인증 완료 후 **업로드 화면으로 직행**.
- **전환 트리거:** 선택지를 늘리지 않는다. Google 단일 버튼 + "왜 Google인가"(비밀번호 불필요·1클릭)만. 이메일/비번 폼은 MVP 범위 밖.
- **적용 원칙:** Progressive Disclosure(가입은 그 자체가 목적이 아니라 업로드로 가는 통로 — 가입 직후 곧장 업로드), Consistency(여기서도 1차 행동은 accent pill).
- **마찰 제거:** 로그인 후 빈 대시보드로 떨어뜨리지 않는다. **업로드 패널이 이미 열린 상태**로 진입시켜 "다음에 뭘 하지?"를 없앤다.
- **검수 체크:**
  - [ ] OAuth 콜백 실패 시 명확한 한국어 에러 + 재시도 경로.
  - [ ] 로그인 직후 랜딩 N초 + 곧장 `/dashboard` 업로드 상태.

### 단계 3 — 첫 업로드 → Aha 순간 (가장 중요한 활성화 지점)
- **단 하나의 목표:** CSV 한 개를 끝까지 업로드해 **첫 분석 결과를 본다.** (활성화의 정의)
- **전환 트리거:** 업로드 직후 결과가 빠르고 구체적으로 뜨는 경험. "내 돈이 이렇게 쓰였구나"를 처음 보는 순간이 리텐션을 결정한다.
- **적용 원칙:**
  - **Progressive Disclosure:** `파일 드롭 → (실패 시에만)컬럼 매핑 1회 확인 → 분석 중 → 결과`. 표준 파서 성공 시 매핑 단계는 보이지 않는다.
  - **Hierarchy:** 결과가 뜨면 결론 숫자(이번 달 지출·전월 대비)를 가장 크게.
  - **Accessibility/Feedback:** 파싱·분석은 시간이 걸리므로 단계별 진행 표시(업로드됨 → 파싱 → 분석 중). Opus는 latency가 크므로(timeout 30s) **규칙·통계 결과를 먼저 렌더하고 AI 인사이트는 뒤늦게 채운다**(스켈레톤 → 채움). AI 실패해도 규칙·통계 결과는 남는다.
- **마찰 제거:**
  - 빈 상태(empty state)에 **샘플 CSV 다운로드/체험** 링크 — "올릴 파일이 없어요"를 막는다.
  - 컬럼 매핑 확인 화면은 추론된 매핑을 **기본 채움**해 두고 사용자는 확인만(타이핑 최소화).
  - 인코딩·통화기호·괄호음수는 사용자에게 묻지 않고 `lib`에서 정규화(CLAUDE.md).
- **검수 체크:**
  - [ ] 업로드 → 첫 숫자까지 클릭/대기 단계가 최소인가.
  - [ ] 표준 파서 성공 경로에서 LLM 확인 단계가 **뜨지 않는다.**
  - [ ] 분석 지연 동안 화면이 비어 보이지 않는다(스켈레톤·진행 텍스트).
  - [ ] AI 실패 시에도 규칙·통계 카드가 정상 표시되고, 인사이트 영역만 `unavailable` 안내.

### 단계 4 — Free 결과 (Free 사용자: "쓸 만하네. 더 깊은 건?")
- **단 하나의 목표:** Free 가치를 충분히 체감시켜 신뢰를 쌓고, 자연스럽게 Pro의 깊이를 **궁금하게** 만든다.
- **전환 트리거:** Free를 절름발이로 만들지 않는다. Free가 충분히 좋아야 Pro를 믿는다. 단, Pro만 답할 수 있는 질문("그래서 어디서 줄여야 하나?")을 결과 안에서 자연스럽게 띄운다.
- **적용 원칙:** Hierarchy(StatCard → CategoryDonut/TrendLine → AnomalyList → InsightsPanel 순), Consistency(차트 색 = 토큰: 추이 라인 `--primary`, 이상치 `--semantic-down`, 절약 `--semantic-up`), Proximity(각 차트 + 범례 + 합계를 한 카드에).
- **검수 체크:**
  - [ ] Free 결과만으로도 "이번 달 지출 구조"라는 약속이 완결되는가.
  - [ ] Sonnet 요약(`InsightsPanel`)이 데이터와 모순되지 않는가(부호·합계 정합).

### 단계 5 — Pro 미리보기 (잠금 + 티저)
- **단 하나의 목표:** "이 한 칸을 열고 싶다"는 욕구 생성 → Pro CTA 클릭.
- **전환 트리거:** *보여주되 막는다.* Pro 인사이트 영역을 **blur 처리한 실제 결과 형태**로 노출(빈 자물쇠 아이콘만 두지 않는다). 사용자가 "내 데이터에 대한 진짜 답이 저 안에 있다"고 느껴야 한다.
- **적용 원칙:**
  - **Progressive Disclosure:** Free에서 못 푼 질문을 Pro 카드에서 받는다.
  - **Contrast:** 잠금 카드의 잠금 해제 버튼만 accent pill. 주변은 차분하게.
  - **Honesty(보이스):** 과장·느낌표 없이 "Pro는 가맹점 단위로 절약 여지를 더 깊게 봅니다" 같은 평서체.
- **게이팅 UX:** 서버 DB 구독상태로만 판정(CLAUDE.md). 미구독·쿼터 소진 시에도 **402가 아니라 200 + Free 결과 + `pro.status=locked|unavailable`**. 따라서 잠금은 "에러"가 아니라 화면의 **정상 상태**로 디자인한다.
  - `locked`(미구독): "Pro로 잠금 해제" CTA.
  - `unavailable`(쿼터 소진·timeout·실패): "잠시 후 다시" 톤 — 결제 유도가 아니라 일시적 상태임을 구분해 표기. 결제 CTA를 띄우지 않는다.
- **검수 체크:**
  - [ ] 잠금 카드가 실제 결과의 형태(blur)로 가치를 암시한다.
  - [ ] `locked`와 `unavailable`의 메시지·CTA가 명확히 다르다.
  - [ ] Free 결과는 잠금과 무관하게 항상 완전히 보인다.

### 단계 6 — 결제 (Polar 체크아웃)
- **단 하나의 목표:** 결제 완료.
- **전환 트리거:** 결정 직전에 새로운 정보로 망설이게 하지 않는다. 가격·포함 항목을 미리보기 단계에서 이미 보여줬으므로, 체크아웃은 **확인과 결제만**.
- **적용 원칙:** Progressive Disclosure(결제 폼은 마지막에만), Consistency(가격 표기 = mono, 통화 명시), Proximity(가격 + 포함 혜택 + 결제 버튼을 한 묶음).
- **신뢰/보안:** `customerExternalId`는 클라이언트 입력이 아니라 서버 세션 `getUser().id`로 강제(CLAUDE.md) — 사용자에게는 보이지 않지만, 결제 실패/중복을 막아 UX 신뢰로 이어진다. Polar = Merchant of Record임을 (세금·환불 주체) 한 줄로 안내해 해외 결제 불안을 낮춘다.
- **검수 체크:**
  - [ ] 결제 진입 전 최종 금액·주기·통화가 mono로 명확.
  - [ ] 결제 취소/실패 시 Free 상태로 깔끔히 복귀(잔여 잠금 상태 깨지지 않음).

### 단계 7 — Pro 활성 (잠금 해제 직후)
- **단 하나의 목표:** 결제가치를 **즉시** 체감(post-purchase Aha). 후회를 막는다.
- **전환 트리거:** 결제 직후 잠겼던 카드가 채워지는 전환을 보여준다. 웹훅 반영 지연이 있으면 "활성화 처리 중" 상태를 명확히(빈 화면 금지).
- **적용 원칙:** Feedback(상태 전환 가시화), Hierarchy(Opus 심층 인사이트를 결과 상단 가까이 승격).
- **검수 체크:**
  - [ ] 결제 → Pro 인사이트 노출까지 빈 구간이 없다(처리 중 상태 포함).
  - [ ] 캐시(`unique(user_id,input_hash)`) 적중 시 재분석 없이 즉시 표시.

### 단계 8 — 재방문 / 리텐션 (월 주기)
- **단 하나의 목표:** 새 명세서 추가를 습관으로.
- **전환 트리거:** 재방문 시 빈 화면이 아니라 **지난 분석 + "이번 달 명세서 추가" 단일 CTA**로 진입. 월 주기 제품이므로 "지난달 대비"가 다음 업로드의 동기다.
- **적용 원칙:** Consistency(업로드 동선이 처음과 동일), Hierarchy(전월 대비 delta를 강조).
- **검수 체크:**
  - [ ] 재방문 시 과거 결과가 보존·표시되는가.
  - [ ] "새 명세서 추가" 동선이 첫 업로드와 동일한 패턴인가.

## 3. 마이크로카피 & 보이스 (전환의 미세 조정)

DESIGN.md의 보이스(차분한 평서체 권위, 노이모지·노느낌표, 한국어 우선)를 전환 카피에도 그대로 적용한다.

- **CTA는 동사 + 결과.** "제출" 대신 "분석 시작", "구매" 대신 "Pro로 잠금 해제", "Free로 시작".
- **상태별 톤 분리:** `locked`는 권유("더 깊게 보기"), `unavailable`은 안심("잠시 후 다시 시도"), `error`는 책임지는 톤 + 재시도. 셋을 섞지 않는다.
- **과장 금지:** "AI가 모든 걸 알려드립니다" 대신 "가맹점 단위로 절약 여지를 정리합니다". 금융 제품에서 과장은 즉시 불신이 된다.
- **숫자는 항상 단위·통화·기간과 함께.** "−18%"가 아니라 "전월 대비 −18%".

## 4. 상태(state) 설계 — 마찰은 빈 상태와 에러에 숨는다

성공 경로만 디자인하면 퍼널은 비어 있는 화면에서 샌다. 각 화면은 아래 상태를 모두 갖는다.

| 상태 | 원칙 | finsight 적용 |
|---|---|---|
| Empty | Progressive Disclosure | 대시보드 최초: 업로드 패널 + 샘플 CSV 체험 링크. "올릴 게 없다"를 막는다. |
| Loading | Feedback/Accessibility | 단계 텍스트(파싱 → 분석 중) + 스켈레톤. 규칙·통계 먼저, AI는 나중에 채움. |
| Partial | Contrast | 규칙·통계 OK + AI `unavailable`: 결과는 그대로, 인사이트 영역만 격리 안내. |
| Locked | Hierarchy | Pro 미리보기 blur + 단일 accent 해제 CTA. |
| Error | Honesty | 한국어 원인 + 재시도. AI 실패가 전체 실패로 보이지 않게 격리. |
| Success | Hierarchy | 결론 숫자를 최상단으로. |

## 6. 출시 전 퍼널 검수 체크리스트 (요약본)

- [ ] 모든 화면에 1차 CTA(accent pill)가 **정확히 하나**.
- [ ] 랜딩: 가입 전 샘플 데모로 가치 증명, 보안 섹션이 가격보다 앞.
- [ ] 가입 직후 곧장 업로드 상태로 진입(빈 대시보드 금지).
- [ ] 표준 파서 성공 시 컬럼 매핑 단계 비노출(마찰 0 경로).
- [ ] 분석 지연 동안 스켈레톤/진행 텍스트, 규칙·통계 먼저 렌더.
- [ ] AI 실패가 전체 실패로 번지지 않음(`unavailable` 격리).
- [ ] Pro 잠금은 에러가 아닌 정상 상태(200), `locked`≠`unavailable` 메시지 분리.
- [ ] 결제 직전 새 정보로 망설이게 하지 않음(가격은 미리보기에서 노출 완료).
- [ ] 결제 직후 잠금 카드가 채워지는 전환 가시화.
- [ ] 재방문 시 과거 결과 + 동일 업로드 동선.
- [ ] 숫자 전부 mono, 색은 토큰만, 'Vantage' 노출 0.
- [ ] 키보드 도달·포커스·aria·색 외 의미 전달(기호 병기) 확인.

## 7. 측정 — 퍼널 누수 추적 (Post-MVP 관측 파이프라인 대비)

각 단계 전환율을 별도 이벤트로 본다(MVP는 정의만, 파이프라인은 Post-MVP).

```
랜딩 방문 → 데모 조회율 → 가입 전환율 → 첫 업로드 완료율(=활성화)
 → Free 결과 도달율 → Pro 미리보기 조회율 → 결제 시작율 → 결제 완료율
 → Pro 인사이트 조회율 → 30일 재방문율
```

- **가장 중요한 한 지표:** 첫 업로드 완료율(활성화). 여기가 막히면 그 뒤 모든 지표가 무의미하다.
- 단계별로 "직전 단계 대비" 전환을 본다. 가장 큰 drop이 다음 개선 대상.
