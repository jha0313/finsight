finsight · 하네스 Eval

우리 코드 검사기, 누가 검사하지?

우리는 코드를 검사하는 도구(리뷰 하네스)를 만들었어요. 그런데 그 도구 자체가 잘 작동하는지는 누가 확인할까요? 이 Eval이 바로 그 검사기를 시험하는 "쪽지시험"입니다.

코드엔 테스트가 있다면, 하네스엔 Eval이 있다.

CLAUDE.md 한 줄, 프롬프트 한 문장만 바꿔도 AI 도구의 행동은 통째로 달라질 수 있어요. 코드가 회귀하듯, 하네스도 회귀합니다. Eval은 그 회귀를 잡는 안전망이에요.

1받아쓰기 시험으로 비유하면

등장인물은 딱 넷이에요. 이 넷만 알면 끝.

Golden Set

정답지

문제(코드)와 정답(이건 위반 / 이건 정상)이 미리 적힌 시험지 5장.

🧑‍🎓
Reviewer

학생

시험을 보는 사람 = 우리가 만든 리뷰 도구. 코드를 보고 "룰 위반!"을 찾아냅니다.

🧑‍🏫
Judge

채점 선생님

학생 답을 정답지와 대조해 맞음/틀림을 매깁니다. 학생과는 다른 사람(다른 모델).

🚦
Gate · run.ts

성적표 & 버저

하나라도 틀리면 빨간불(버저). 배포·머지를 막습니다.

2전체 흐름도

정답지 한 장이 왼쪽에서 들어가 오른쪽 성적표로 나옵니다. 이걸 5장 반복해요.

📋cases/*.md

정답지 한 장

문제 + 정답 라벨이 같이 들어있어요
  • 위에: expect (violation/pass)
  • 아래: 검사할 코드 조각
  • 위반 4장 + 정상 1장
→
🧑‍🎓Sonnet

학생이 푼다

리뷰어가 코드를 읽고 리뷰를 씀
  • CLAUDE.md 룰을 기준으로
  • "이 코드 위반? 정상?"
  • = 우리가 시험하는 대상
→
🧑‍🏫Opus

선생님이 채점

학생 답 ↔ 정답 라벨 대조
  • 기대대로면 → pass
  • 놓치거나 오탐 → fail
  • 이유(reasoning)도 함께
→
🚦run.ts

성적표

5장 모아서 최종 판정
  • 전부 pass → exit 0 🟢
  • 하나라도 fail → exit 1 🔴
  • = 회귀 게이트
🔁 정답지는 자란다. 운영하다 새로 발견한 실패를 정답지(cases/)에 한 장씩 추가해요. 테스트 스위트가 버그 하나당 한 줄씩 늘듯이 — 5장으로 시작해서 천천히.

3정답지 한 장은 이렇게 생겼다

왼쪽이 실제 시험지 한 장(케이스 1). 오른쪽이 그게 흘러가는 모습.

cases/01-lib-imports-sdk.md정답지
--- 정답 라벨 ---
id: 01-lib-imports-sdk
rule: lib는 외부 SDK를 import 금지
expect: violation   ← 정답
severity: critical
--- 검사할 코드 ---
// src/lib/analysis/summarize.ts
+import Anthropic from "@anthropic-ai/sdk";
+const client = new Anthropic({ ... });
🧑‍🎓 학생(리뷰어) 답

"lib/가 외부 SDK를 직접 import하네요. 레이어 의존성 룰 위반, critical입니다."

🧑‍🏫 선생님(judge) 채점

정답은 violation. 학생이 위반을 정확히 critical로 잡았다 → PASS ✓

4돌리면 두 가지 결과

실제로 셋업하고 둘 다 확인했어요 — 통과도, 고장도 제대로 잡는지까지.

🟢 정상 — 5/5 통과 · exit 0
✓ 01 lib-imports-sdk
✓ 02 money-parsefloat
✓ 03 service-role-public-env
✓ 04 pro-gating-request-body
✓ 05 lazy-provider-injection (정상)

초록불. 리뷰 도구가 위반 4개를 잡고, 정상 코드는 오탐 안 함. 배포 OK.

🔴 회귀 시뮬 — 4/5 · exit 1
✗ 01 lib-imports-sdk  ← 일부러 정답을 틀리게
✓ 02 money-parsefloat
✓ 03 service-role-public-env
✓ 04 pro-gating-request-body
✓ 05 lazy-provider-injection

빨간불. 케이스 1의 정답을 일부러 뒤집자 선생님이 "오탐"으로 잡아내 exit 1. 버저가 진짜 울린다는 증명.

6왜 명령어를 둘로 나눴나

npm test  — 공짜·항상

정답지가 5장 맞는지(위반4+정상1), 읽고 집계하는 함수가 멀쩡한지만 검사. AI를 안 부르니 키도 돈도 필요 없고 CI가 매번 돌려요.

$ npm test  → 367 passed

npm run eval  — 진짜 시험

학생·선생님(Claude)을 실제로 불러 5장을 채점. 키·비용·시간이 들고 결과가 조금 흔들려서, 평소 그린 게이트와 떼어놨어요.

$ npm run eval  → 5/5 통과
