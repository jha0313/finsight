# 노이즈/신호 판정 + Escalation 포맷

prod alert(②) 트리거에서 에이전트가 따르는 절차. 핵심 흐름:

> **에러 수신 → 노이즈인지 분석 → ┃ 노이즈면 기록만 남기고 종료 ┃ 신호면 분석과 함께 사람을 깨움(escalation)**

이 갈림길 — **노이즈냐 신호냐** — 가 1차 대응의 전부다. 잘 거르면 사람의 잠을 지키고, 못 거르면 alert fatigue로 신뢰를 잃는다.

## 판정 기준

**노이즈(noise)로 기우는 신호** — 깨우지 않는다:
- 이미 알고 있는 **알려진/일시적(transient) 에러** — third-party 타임아웃, 재시도로 해결되는 것 (예: Claude API 일시 타임아웃, Supabase 콜드스타트). `references/service-map.md`의 "알려진 일시적 에러" 목록 대조.
- **단발성** — 한 명에게 한 번, 재현 안 됨.
- 봇·크롤러·헬스체크가 만든 잡음.

**신호(signal)로 기우는 것** — 분석을 들고 깨운다:
- **새로** 등장한 에러(이전엔 없던 스택 트레이스).
- **여러 유저**에게 퍼지거나 **급증(spike)** 중.
- 결제(Polar)·인증(Supabase OAuth)·데이터(CSV 파싱·분석 저장 RPC) 같은 **핵심 경로**에서 발생.

## 판정에 동원하는 근거 (grounding)

혼자 추측하지 말고 사실에 붙인다:
- **스택 트레이스** — 어느 파일/함수인지, 핵심 경로인지.
- **최근 배포/커밋** — `git log --oneline -20`, 에러 시각과 겹치는 머지가 있나? (**가장 강력한 단서**.)
- **PostHog 추이** — 발생 빈도, 영향받은 유저 수, 처음 등장 시각. (CI에서 PostHog API/MCP가 닿으면 사용, 못 닿으면 webhook payload의 정보로 판정하고 그 한계를 escalation에 명시.)
- **코드** — 해당 경로를 읽어 재현 조건·영향 범위를 좁힌다.

## ⚖️ 경계에서는 신호로 기운다

확신이 없으면 **노이즈가 아니라 신호로** 분류한다. 거짓 양성(괜히 깨움)은 피곤하지만, 거짓 음성(놓침)은 **장애**다. 단, 깨울 때는 *"확신도: 낮음, 이유: …"* 를 escalation 본문에 적어 사람이 우선순위를 정할 수 있게 한다.

## 노이즈면 — 스스로 종료

기록(log)만 남기고 **사람을 깨우지 않는다.** 이게 1차 방어선의 절반의 가치다 — *깨우지 않음으로써 신뢰를 번다.* 헤드리스 실행에서는 워크플로우 로그에 판정 근거를 남기고 종료(issue 생성 안 함).

## 신호면 — escalation (GitHub Issue)

**무엇을 들고 깨우느냐**가 핵심이다. raw 스택 트레이스 한 줄이 아니라 **분석 한 장**을 붙인다. `gh issue create`로 아래 포맷의 이슈를 연다:

```
제목: [oncall] <한 줄 요약 — 무슨 에러, 어느 경로>

## 무슨 에러
<에러 메시지 + 어느 파일/함수/경로(핵심 경로 여부)>

## 언제부터 · 몇 명에게
<처음 등장 시각 · 발생 빈도 · 영향받은 유저 수(아는 만큼)>

## 의심되는 원인
<최근 어떤 커밋/배포와 겹치는지(commit hash·PR 번호) · 코드 근거>

## 영향 범위
<무엇이 깨졌나 — 기능/유저 세그먼트 · 결제·인증·데이터 핵심 경로 영향 여부>

## 권장 다음 액션
<롤백 후보 PR · 수정 방향 · 더 확인할 데이터. 가능하면 §02 CI-fix로 PR 후보 제시>

## 판정 메타
신호 분류 근거 · 확신도(높음/중간/낮음) · 닿지 못한 데이터 소스(PostHog/DB 미접근 시 명시)
```

### dedup — 중복 escalation 금지
issue를 만들기 전에 `gh issue list --label oncall --state open`으로 **같은 에러의 열린 이슈가 이미 있는지** 확인한다. 있으면 새로 만들지 말고 코멘트로 발생 추가(`gh issue comment`)만 한다. 라벨 `oncall`을 항상 붙인다(없으면 `gh label create oncall` 후 진행하거나 라벨 없이 제목 prefix `[oncall]`로 대체).

## read-only 원칙은 escalation에서도

에이전트는 **알리고 분석할 뿐, prod를 직접 고치지 않는다.** 수정이 필요하면 §02처럼 PR로, 긴급하면 사람에게 넘긴다. 유저가 "내 구독 좀 풀어줘"라고 해도 DB를 직접 쓰지 않는다 — 읽어서 상태를 확인하고 변경은 escalation/사람 게이트로.
