# CI 실패 → 분석 → 수정 → PR (트리거 ①)

`ci.yml`(lint+build+test)이 실패하면 `oncall-ci-fix.yml`이 깨어나 이 절차를 따른다. **Hermes 코딩 자동화(GitHub 이슈=사람의 의도)와의 차이는 트리거 하나** — 여기선 사람의 의도 없이 **시스템이 깨진 사실 자체**가 트리거다(요청 처리가 아니라 사고 대응).

## 흐름 (네 단계)

> **빌드 실패 감지 → 로그 분석 → 코드 수정 → PR 생성**

### ① 컨텍스트 수집
에이전트가 받는 것: ⓐ 실패 로그(스택 트레이스·에러 메시지), ⓑ repo 하네스(CLAUDE.md·코드 컨벤션), ⓒ 깨진 코드. 이 셋이면 "왜 깨졌는가"를 사람만큼 좁힌다.
- **실패한 잡의 로그만** 본다. 성공 잡 로그까지 넣으면 컨텍스트가 노이즈로 찬다. 실패 스텝 로그 + 변경된 파일 diff(`git diff`)로 좁힌다.

### ② 근본 원인 분석
단순 타입 에러인지, 누락된 import인지, 테스트가 코드 변경을 못 따라온 건지 짚는다. `CLAUDE.md`의 CRITICAL 규칙을 위반하는 수정은 하지 않는다(금액 float 금지, lib 외부 SDK import 금지 등 — `references/service-map.md`).

### ③ 수정 브랜치
**원본 브랜치/main에 직접 머지하지 않는다.** 새 브랜치(`oncall/fix-<run_id>` 권장)에 패치를 담는다.

### ④ PR 생성 — 사람 게이트
`gh pr create`로 수정 PR을 연다. PR 본문엔 **무엇이 왜 깨졌고 어떻게 고쳤는지** 분석을 담는다(사람은 빨간 X + raw 로그가 아니라 원인 분석 + 패치 한 세트를 받는다). 이 PR은 `code-review.yml`이 자동 리뷰하고 **최종 머지는 사람**이 한다 — 루프가 닫힌다.

## 함정 (CRITICAL)

- **출력은 머지가 아니라 PR** — push 권한을 줘도 **자동 머지 절대 금지**. "고쳤다"와 "반영했다" 사이에 사람을 한 명 세운다.
- **봇 PR 무한루프 가드** — 에이전트가 연 PR이 또 CI를 돌려 또 실패하면 또 깨운다. 워크플로우에서 **봇/oncall 브랜치가 트리거한 실패는 skip**(actor·브랜치 prefix 체크) + 재시도 상한. 자율 시스템의 1순위 사고.
- **secret 노출** — 로그에 토큰/키가 찍힐 수 있다. PR 본문으로 새면 사고. GitHub가 등록 secret을 자동 마스킹하지만, **에이전트는 로그 원문을 PR 본문에 그대로 붙이지 말고**, 분석한 요약만 쓴다. secret 패턴(키=값, Bearer 토큰)이 보이면 제거한다.
- **하나의 사고에 PR 하나** — 같은 run에 대해 PR을 여러 개 만들지 않는다. 기존 oncall PR이 열려 있으면 갱신.
