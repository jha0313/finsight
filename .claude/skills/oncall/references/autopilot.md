# Autopilot — 유저 질의에 근거로 답하기 (트리거 ③)

> 한 줄: 유저(제품)·내부(운영) 질문에 **코드·로그(PostHog)·DB(Supabase read-only)·git log** 근거로 답한다. 지어내지 않고, **유저로 직접 나가는 답은 draft → 사람 승인.**

이건 oncall의 세 번째 입구다. ①CI-fix·②triage가 *시스템이 던지는* 이벤트라면, ③Autopilot은 *사람이 던지는 질문*이다. 답하는 두뇌는 이 하네스로 이미 충분하다 — 빠진 건 능력이 아니라 **항상 켜진 입구**(인앱 티켓 수신·세션·발송 채널). 그건 `docs/ONCALL_SETUP.md`에 **분리된 후속**으로 설계만 둔다. 지금은 **로컬 one-shot**으로 쓴다 (예: *"방금 들어온 이 에러, 유저한텐 뭐라고 답하지?"*).

## 두 청중 — 먼저 누구에게 답하는지 가른다 (CRITICAL)

| 축 | **외부 유저** (제품 도우미) | **내부 운영 동료** (팀) |
|---|---|---|
| **채널(지금)** | 인앱 지원 티켓 페이로드(질문 텍스트, 가능하면 `user_id`) | 터미널 one-shot |
| **출력** | **draft만** → 사람 승인 후 발송 | 바로 답(승인 게이트 없음 — 내부) |
| **톤** | 비기술·친절·한국어, 결론 먼저 | 기술적, `file:line`·커밋·근거 링크 |
| **노출 금지** | 스택트레이스·내부용어·다른 유저 데이터·PII·`service_role` | (내부엔 근거 전부 노출 OK) |

**청중 판별:** 입력에 명시한다 — 유저 티켓은 `[유저티켓]`/페이로드 형태로, 내부 질문은 터미널에서 직접. **모호하면 물어본다**(혼자 정하지 말 것). 둘 다 필요할 수 있다 — 같은 에러에 *"유저한텐 뭐라 답하지"*(외부 draft) + *"원인 뭐야"*(내부 분석)를 동시에 낼 수 있다.

## 근거(grounding) — 지어내지 않는다 (CRITICAL)

운영에서 가장 위험한 건 **그럴듯한 답**이다. 답의 **모든 주장**은 아래 사실 소스에 붙인다. 소스는 `references/service-map.md`와 동일:

- **코드베이스** — repo에서 동작을 직접 Read. "정상이 무엇인지"는 service-map의 핵심 경로 표.
- **로그/에러 (PostHog)** — 그 에러가 실제로 났는지·몇 명에게·추이가 어떤지. 닿으면 빈도/영향 유저 확인, **못 닿으면 그 한계를 답에 명시**.
- **DB (Supabase, read-only)** — 구독 상태 등 사실을 **읽기만** 확인. `?read_only=true`. **쓰기 금지.**
- **최근 배포/커밋** — `git log --oneline -20`, `git show <hash>`. main 머지 ≈ 배포. "최근 변경과 겹치나?"가 원인 좁히기의 가장 강한 단서.

규칙:
- **확인 못한 건 단정하지 않는다.** "확인 못함"으로 명시하거나 추가 확인을 제안한다.
- 알려진 일시적 에러(service-map의 목록 — Opus 타임아웃·max_tokens 잘림·Supabase 콜드스타트 등)는 그 맥락으로 설명하되, **급증·지속·여러 유저면** 신호로 보고 ②triage/escalation으로 넘긴다.

## 외부 유저 답 — draft → 승인 (CRITICAL)

1. **자동 발송 절대 금지.** 언제나 **초안만** 만든다. 발송은 사람이 승인 후. (always-on 발송 채널은 후속.)
2. 입력: 인앱 티켓 페이로드(질문 텍스트 + 가능하면 `user_id`).
3. read-only로 사실 확인 → 유저 친화 답 초안. 내부 근거는 **별도 블록**에 담아 발송 본문과 섞지 않는다.
4. **구독/결제 변경은 직접 하지 않는다** — read-only 게이트가 여기서도 그대로다. 유저가 *"구독 좀 풀어줘"* 라고 해도 **DB를 쓰지 않는다**: 상태를 **읽어** 확인하고, 변경이 필요하면 **escalation(사람/PR 게이트)**. (`references/triage-rubric.md`의 read-only 원칙과 동일.)
5. **노출 금지:** 카드/계좌번호, 내부 스택트레이스, 다른 유저 데이터, `service_role`/내부 인프라. 마스킹·PII 규칙은 `CLAUDE.md` CRITICAL을 따른다.

## 내부 운영 동료 답 — 터미널 one-shot

- 승인 게이트 없이 바로 답하되, **근거는 외부보다 더 빡빡하게** 붙인다(`file:line`·commit·PostHog·DB read-only).
- 전형: *"어젯밤 prod 왜 났어"* → `git log`로 그 시각과 겹치는 머지 → PostHog 추이 → 코드 경로 Read로 재현 조건 좁힘.
- ②triage와 연속선상이다 — 내부 답이 "이건 신호다"로 기울면 그대로 escalation 포맷으로 넘긴다.

## 출력 포맷

**외부 유저 draft** (발송 본문 / 내부 근거를 분리):
```
[DRAFT — 사람 승인 후 발송 · 자동 발송 금지]
대상: 인앱 티켓 #<id> (user_id: <…>)

--- 발송 본문(유저용) ---
<결론 먼저. 비기술·친절·한국어. 스택트레이스/내부용어 없음>

--- 내부 근거(발송 X) ---
- 코드: <file:line>
- 로그(PostHog): <빈도·영향 유저 / 못 닿으면 "미접근">
- DB(read-only): <구독 상태 등 / "쓰기 안 함">
- 배포/커밋: <hash·PR>
확인 못한 것: <…>
승인 전 점검: 변경(구독/결제) 요청이면 escalation 필요한지
```

**내부 답**:
```
## 답
<결론>
## 근거
- 코드: <file:line>
- 배포: <commit/PR — 에러 시각과 겹침 여부>
- 로그(PostHog): <추이·영향 유저 / 미접근 시 명시>
- DB(read-only): <확인한 사실 / 쓰기 안 함>
## 확신도 / 못 닿은 소스
<높음·중간·낮음 · 미접근 소스 명시>
```

## 못 끝낼 일 → escalation

read-only 안에서 답이 끝나지 않으면 빈손으로 두지 말고 넘긴다:
- **구독/결제/계정 상태를 바꿔야** 하는 요청 → 직접 못 함 → `references/triage-rubric.md`의 escalation(GitHub Issue) 또는 사람에게.
- **코드 수정이 필요** → 트리거 ①(CI-fix) 패턴으로 분석 동반 **PR**.
- **확신이 낮음** → 답하되 "확신도: 낮음, 이유: …"를 명시(외부엔 발송 보류 권고, 내부엔 그대로 표기).

## 한계 (지금 / 후속)

- **지금:** 로컬 one-shot. 질문(또는 티켓 페이로드)을 주면 grounding 붙은 답/초안을 낸다.
- **후속(always-on, 분리):** 인앱 티켓 수신 → 검증·멱등 → 헤드리스 에이전트 → **draft를 지원 인박스에 저장**(사람 승인 발송). ②triage와 동형 구조. 설계는 `docs/ONCALL_SETUP.md`.

## 워크드 예시 (one-shot 시연)

근거는 모두 repo 사실에 붙어 있다(service-map 핵심 경로 표·알려진 일시적 에러 목록).

### 예시 A — 외부 유저 티켓: "Pro 결제했는데 AI 인사이트가 안 떠요"
입력(페이로드):
```json
{ "channel": "in_app_ticket", "id": 4821, "user_id": "u_…", "text": "Pro 결제했는데 분석은 나오는데 AI 인사이트만 비어 있어요" }
```
grounding 절차:
1. **DB(read-only)** — 구독 상태 확인. `status=active AND current_period_end > now()`이면 게이팅 문제 아님(미구독/만료면 게이팅 — 그땐 갱신 안내, **DB 쓰기로 풀어주지 않음**).
2. **service-map** — 구독은 정상인데 인사이트만 빈 경우, 알려진 일시적 에러 후보: **Opus(`claude-opus-4-8`) 타임아웃** 또는 **structured-output max_tokens 잘림** → 설계상 규칙·통계 결과는 보존되고 AI 인사이트만 `unavailable`(200). 코드 경로: `src/app/api/analyze/route.ts`·`lib/orchestration`.
3. **PostHog** — 같은 증상이 **급증/여러 유저**면 단발 노이즈가 아니라 신호 → ②triage/escalation으로 승격. 못 닿으면 그 한계 명시.

draft 출력:
```
[DRAFT — 사람 승인 후 발송 · 자동 발송 금지]
대상: 인앱 티켓 #4821 (user_id: u_…)

--- 발송 본문(유저용) ---
결제는 정상 처리됐고 Pro로 잘 적용돼 있어요(분석이 나오는 게 그 증거예요).
AI 인사이트만 비어 보이는 건 심층 분석 단계가 일시적으로 지연되면 그 부분만
건너뛰고 결과를 먼저 보여주기 때문이에요. 같은 명세서로 한 번 더 분석을
돌려보시면 인사이트까지 채워질 가능성이 높습니다. 그래도 비어 있으면 이 티켓에
회신 주세요 — 바로 더 살펴볼게요.

--- 내부 근거(발송 X) ---
- DB(read-only): 구독 active, current_period_end 미래 → 게이팅 아님(쓰기 안 함)
- 코드: src/app/api/analyze/route.ts → 규칙·통계 보존, AI만 unavailable로 격리
- service-map: Opus 타임아웃 / max_tokens 잘림(알려진 일시적)
- 로그(PostHog): [급증 여부 확인 / 미접근이면 명시]
확인 못한 것: 이 user_id의 해당 analyze 호출이 타임아웃인지 잘림인지(로그 핑거프린팅 필요)
승인 전 점검: 구독 변경 요청 아님 → escalation 불필요. 단 PostHog에서 급증이면 ②로 승격.
```

### 예시 B — 내부 운영 동료(터미널): "어젯밤 analyze에서 500 올라왔는데 뭐야"
grounding 절차: `git log --oneline -20`으로 어젯밤 머지 확인 → PostHog로 그 에러 추이·영향 유저 → `src/app/api/analyze/route.ts`·관련 `lib/` 코드 경로 Read.
출력:
```
## 답
analyze 500은 [원인 가설]. [최근 머지 X / 알려진 일시적 Y]와 겹침.
## 근거
- 배포: <hash> (어젯밤 머지 — 에러 시각과 겹침/안 겹침)
- 코드: src/app/api/analyze/route.ts:<line>
- 로그(PostHog): <빈도·영향 유저 / 미접근 명시>
- DB(read-only): <관련 사실 / 쓰기 안 함>
## 확신도 / 못 닿은 소스
중간 — PostHog 추이 미접근 시 명시. 급증이면 ②triage로 escalation.
```
